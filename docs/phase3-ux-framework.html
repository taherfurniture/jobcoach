Navigation structure

Primary nav (5 items): Dashboard · Career Profile · Coach · Jobs · Settings — identical set on every device, so muscle memory transfers between mobile and desktop.

Mobile: bottom tab bar, five icons, current section highlighted.
Desktop: left sidebar, same five items, more label space.

Coach is not confined to its own tab. A persistent quick-access affordance (docked panel on desktop, floating action button on mobile) reaches the coach from any screen — carrying context from wherever the user was (e.g. "ask coach about this job" from a job details screen opens Coach pre-loaded with that job). This is a structural decision, not a visual one: the coach is meant to feel woven through the product, not parked in a single chat screen.

Job comparison MVP — lightweight

  1. From Saved Jobs, select 2–3 listings
  2. View side-by-side: score, salary (if available), location, work model, top missing qualifications
  3. No dedicated comparison engine at MVP — this is a display transform of already-saved data, not new computation

Coach-narrated comparison ("which of these actually fits you better, and why") is a natural post-MVP extension once this pattern is validated.

Application tracking post-MVP

  1. User manually sets a status per saved job: Not applied / Applied / Interviewing / Offer / Rejected
  2. Optional notes and follow-up reminders
  3. No automatic submission, ever — this logs what the user did outside the app

Course recommendations post-MVP

  1. Triggered from a skill gap surfaced in a coach conversation or job match
  2. Shows curated provider search links per gap (link-out, per the Phase 1 strategy)
  3. User manually marks a resource "started" / "completed" — no LMS integration at this stage

03 Extraction Review & Correction

Purpose
Turn AI-extracted text into a confirmed profile — the core trust moment of the product
Primary action
Review each field, confirm or correct it
Secondary actions
Add a missed field; re-upload a different file

Structured extraction, every field pre-tagged Confirmed/Inferred/Missing; nothing writes to the permanent profile until this screen completes

Trust indicators
Visible per-field tag (e.g. "AI-inferred — please confirm")

Failure

Part D — Behavior

States, devices, accessibility

State patterns

Empty

Always explains why it's empty and gives one clear next action — e.g. "No job matches yet — complete your profile to see matches," never a bare "nothing here."

Loading

Skeleton screens for lists (jobs, profile fields); a distinct "thinking" indicator for Claude responses, since AI latency reads differently to users than network latency.

Error

Names what went wrong in plain language and offers a next step (retry, manual entry, contact) — never a raw error code shown to the user.

Permission

When a feature needs a consent category not yet granted, shows what's needed and why inline, with a direct link to grant it — never silently disabled without explanation.

Mobile vs. desktop

Mobile: bottom tab bar (5 items), single-column layouts, coach reachable via a floating action button, native file picker/camera for document capture.

Desktop: left sidebar nav, multi-column layouts where useful (e.g. profile categories as a two-pane view), coach reachable via a docked panel, drag-and-drop upload.

Both: RTL mirroring for Arabic applies to layout direction, not just text — nav position, directional icons (back arrows), and form field order all flip together.

Accessibility considerations

  • A full keyboard path through every flow above, especially Upload → Review → Interview
  • A field's Confirmed/Inferred state is announced to screen readers, not conveyed by color alone
  • The coach chat interface is screen-reader usable — message roles and loading state are announced
  • Touch targets on the mobile tab bar and job-card actions meet minimum size guidance
  • RTL layout tested with a screen reader in Arabic, not just checked visually
