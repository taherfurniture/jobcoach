Part C — Engineering

Repo, branches, environments, standards

GitHub repository structure

/app # Next.js routes — one folder per screen in the Phase 3 sitemap /components # UI components, matching the Phase 4 design system 1:1 /lib /claude # Anthropic SDK calls: extract, interview, coach, score /prompts # versioned prompt files — changes are diffable and reviewable /supabase # typed DB client, RLS-aware queries /supabase/migrations # schema history, applied via Supabase CLI /tests # extraction test set, RLS tests, critical-path tests /docs # this framework, saved as the project's living reference .github/workflows # CI: lint, typecheck, test on every PR

Branching & environments

Branching: main is production, always deployable. Every task gets a short-lived feature/* branch, merged via PR. No long-lived develop branch — with one builder (you + AI), that extra layer is pure overhead, not safety.

Environments: Local dev and PR previews both point at a separate dev Supabase project and a low-spend dev Claude key — production data and production AI spend are never touched by a branch that isn't main.

Coding standards

TypeScript strict mode, ESLint + Prettier configured from Sprint 1 so every AI-generated file lands consistent. .env.example checked into the repo; real secrets live only in Cloudflare/Supabase environment variables, never in git.

Part D — Quality

Testing, security, acceptance

Testing strategy

Pragmatic, not exhaustive: automated tests concentrate where a silent regression is expensive — auth, Row Level Security policies, provenance-tagging logic, consent enforcement. Everything else is covered by a manual QA checklist run before each production deploy, since 100% coverage isn't the right investment for a 12-week solo build.

AI evaluation strategy

Carried from Phase 2: the fixed multi-format, bilingual CV test set runs before any prompt change ships; real (consented) pilot extractions get spot-checked weekly; a rising per-field correction rate is treated as a direct prompt-quality alarm.

Security testing

  • RLS verification: deliberately attempt cross-user data access in a test, confirm it's denied
  • Dependency scanning via GitHub Dependabot (free, on by default)
  • Secret scanning via GitHub's built-in scanner
  • Prompt-injection check: feed a CV containing an embedded instruction ("ignore previous instructions...") and confirm Claude treats it as inert text, per the Phase 2 sandboxing design

Integration testing

Specifically exercises the third-party seams: Claude API timeout/error handling, Supabase Storage upload/retrieval, Resend delivery, and the job-source ingestion job end-to-end.

User acceptance testing

Before pilot launch, walk the Phase 1 MVP acceptance criteria as a literal checklist — ideally with one or two real test users, not just you. A solo builder can't self-test "does this actually make sense to someone who isn't me."

Part E — Ship & Operate

Deploy, watch, launch, listen

Deployment workflow

GitHub → Cloudflare Workers auto-deploy on merge to main; every PR gets its own preview URL. Supabase migrations apply via CLI before each deploy that changes the schema.

Monitoring plan

Sentry for errors, Cloudflare Web Analytics for traffic. Since there's no ops team, this needs to be a habit, not a dashboard nobody opens: a short weekly check of Claude API spend, Supabase usage %, error rate, and active user count.

Launch checklist

  • All Phase 1 MVP acceptance criteria pass
  • Tested in English and Arabic, mobile and desktop
  • RLS cross-user access denial verified
  • Consent, export, and delete flows tested end-to-end
  • Sentry and usage monitoring live
  • Privacy Policy and Terms of Service published — real legal text, not this framework's privacy section
  • Domain and SSL live
  • AI rate limits configured per user
  • Supabase backups verified restorable

Beta testing plan

Invite 10–20 real users directly, not a public launch. Run for 2–4 weeks. Gather feedback through a short structured form plus a few scheduled calls — enough signal to decide the next iteration without needing a large sample.

Feedback plan

In-app thumbs up/down on AI recommendations (per the Phase 1 analytics plan), a simple feedback email or form, and a weekly review of what's come in.

Post-launch roadmap

Month 4+: expand job sources based on what the pilot actually needed, add application tracking and course recommendations, introduce the paid tier deferred in Phase 1 using real pilot pricing signal, evaluate native mobile only if pilot usage patterns justify it.
