Part A — Stack

The recommendation, and one place it disagrees with you

Everything below is chosen against a single test: can I (Claude Code) build and keep this running correctly without you ever needing to write or debug code yourself. That test consistently favors a small number of managed platforms over best-of-breed pieces stitched together.

Resolved — hosting confirmed on Cloudflare The app deploys to Cloudflare Workers via the OpenNext Cloudflare adapter for Next.js, not Vercel. That adapter is actively maintained but younger than Vercel's native support — budget a little early buffer on first deploy to work through the handful of Node-API or streaming-edge-case issues that commonly show up in a Next.js-on-Cloudflare migration; this framework flags it once here so it isn't a surprise mid-build, not something to keep re-raising. One upside worth noting: hosting, domain, DNS, and CDN now all sit on one Cloudflare account instead of two vendors.

Background / scheduled jobs

Kept deliberately minimal for MVP, all via Cloudflare Cron Triggers: job-source sync (pulls and normalizes new listings on a schedule), preference re-confirmation reminder (periodically prompts users to confirm salary/location/visa/availability haven't changed, per the Phase 1 memory model), stale-listing cleanup (marks expired postings). No dedicated queue service — cron-triggered functions are enough at pilot volume.

Part D — Reliability

What happens when something breaks

Error handling strategy

Error typeUser-facing behavior
Validation (bad input)Inline field-level message, specific about what's wrong
Claude API error/timeoutRetry once with backoff, then "the coach is temporarily unavailable, try again" — never a silent failure or a fabricated response
Auth/session expiredRedirect to login, preserve in-progress form state where feasible
Upload/extraction failureExplicit "couldn't read this file" with a manual-entry fallback offered immediately

Multi-step writes (e.g. extraction → tagging → profile update) run inside a database transaction, so a failure partway through never leaves a half-updated profile.

AI evaluation plan

Because the one hard rule is "never invent a fact," extraction quality needs a real check, not a vibe check: maintain a small fixed set of test CVs (varied formats, English and Arabic) with known-correct structured output, and run it against that set before shipping any prompt change. During the pilot, spot-check a sample of real (consented) extractions weekly. Track a simple signal over time: how often users correct the same field type — a rising correction rate on, say, "job title" is a direct prompt-quality alarm, not something you'd otherwise notice.

Part E — Environments

Dev, deploy, and backup

Development environment

Local Next.js dev server against a separate dev Supabase project (not production data). A separate, low-spend Claude API key for development, distinct from the production key, so a bug in a test loop can't quietly run up production-scale cost.

Deployment & backups

Deployment: GitHub is the source of truth — push to main deploys to production on Cloudflare Workers automatically via the Git integration; every other branch/PR gets its own preview URL, so a change can be checked before it's live. Backups: Supabase's free tier has limited backup retention — budgeting Supabase Pro (~$25/month) is recommended once real user data is involved, specifically for daily backups and point-in-time recovery, not for any feature reason.
